# Mumzworld Companion

A Next.js technical assessment for a bilingual parenting companion. The app builds a local profile for a pregnancy or baby stage, generates grounded stage briefs, answers parenting and shopping questions with tool use, and exposes an eval dashboard for AI quality checks.

Live demo: [mumzworld-ai-companion-dinesh.netlify.app](https://mumzworld-ai-companion-dinesh.netlify.app/)  
Screen Recording: [Video](https://drive.google.com/file/d/1BaYgEY9CtDde71-WSdDx-ypJk5wgeWeR/view?usp=sharing)
Repository: [github.com/Twist-Turn/MumzWorld-Assessment-Dinesh](https://github.com/Twist-Turn/MumzWorld-Assessment-Dinesh)

## Features

| Area | What exists in this repo |
| --- | --- |
| Profile setup | Local onboarding for language, city/currency, pregnancy week, or baby age. Profile state is stored in `localStorage`. |
| Stage dashboard | A personalized brief generated from the local knowledge base and product catalog, with citations and product recommendations. |
| Tool-using chat | `/chat` classifies requests, calls tools such as `search_products`, `lookup_milestone`, `check_safety`, and `build_checklist`, and validates the final JSON response with Zod. |
| Voice flow | `/voice` uses Whisper transcription, then extracts a shopping list from the transcript. |
| Photo safety | `/photo` uses the vision model to assess whether a pictured item appears age-appropriate. |
| Checklists | `/checklists` and `/checklists/[stage]` generate grounded preparation lists for supported stages. |
| Evals | `/evals` and `npm run eval` run the local eval suite over brief/chat behavior, groundedness, refusals, Arabic quality, tool use, voice-style text, and safety cases. |
| Bilingual UI | English/Arabic strings with RTL direction switching for Arabic profiles. |

## Getting Started

### Prerequisites

- Node.js 18+
- An OpenAI API key

### Install

```bash
npm install
cp .env.example .env.local
```

Add your key to `.env.local`:

```env
OPENAI_API_KEY=sk-your-api-key-here
```

### Run Locally

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

## Tooling

### Commands

```bash
npm run dev         # Start the Next.js dev server
npm run build       # Build for production
npm run start       # Start the production server after a build
npm run typecheck   # Run TypeScript checks
npm run embed       # Regenerate data/embeddings.json from KB and products
npm run eval        # Run the AI eval suite and write evals/results.json
```

The `lint` script in `package.json` still points at `next lint`, but this repo does not currently include an ESLint config. Use `npm run typecheck` as the reliable static check unless an ESLint setup is added.

### Embeddings

`scripts/build-embeddings.ts` reads:

- `data/knowledge_base.json`
- `data/products.json`

It embeds each KB entry and product with `text-embedding-3-small`, then writes `data/embeddings.json`. The app loads that file for in-memory cosine similarity search. Regenerate embeddings whenever the KB or product catalog changes:

```bash
npm run embed
```

### Validation

Zod schemas in `src/lib/types.ts` validate profiles, API responses, eval cases, generated AI payloads, tool calls, and eval reports. The chat loop also patches tool trace data into the final response so citations and product recommendations stay grounded even if the model omits fields from its final JSON.

### Agent Tools

The chat agent has four local tools defined in `src/lib/agent/tools.ts`:

| Tool | Purpose |
| --- | --- |
| `search_products` | Search the local catalog by query, category, age, and budget. |
| `lookup_milestone` | Look up pregnancy week or baby month knowledge-base entries. |
| `check_safety` | Check whether a catalog product is age-appropriate and surface safety notes. |
| `build_checklist` | Return starter checklist items for supported preparation stages. |

## Evals

The eval runner is a custom TypeScript script, not a Vitest suite. It is launched by:

```bash
npm run eval
```

The runner:

1. Loads cases from `evals/cases.json`.
2. Calls the real orchestrator functions for `brief`, `chat`, or `checklist` cases.
3. Uses LLM judges in `evals/judges.ts` for groundedness, Arabic quality, refusal correctness, and tool-use appropriateness.
4. Applies local Zod schema validation.
5. Writes a report to `evals/results.json`.

The command exits with a non-zero status if any case fails.

The suite currently covers:

- Pregnancy and baby stage briefs
- Product and milestone grounding
- Arabic responses
- Medical and out-of-scope refusals
- Catalog pricing honesty
- Shopping-list style text inputs
- Safety checks for small toys and swaddling

You can also view and trigger evals from the `/evals` page in the app.

### Eval Philosophy

The suite is focused on the highest-risk behaviors for this product:

- Safety: medical questions must defer to a doctor instead of giving diagnosis or dosage advice.
- Grounding: factual claims should cite a KB entry or catalog product.
- Commerce honesty: product availability and prices must come from `data/products.json`.
- Bilingual quality: Arabic answers should be native-sounding and preserve the same meaning as English behavior.
- Tool use: shopping, milestone, checklist, and safety questions should call the relevant local tools.

## Tradeoffs

### Local JSON Fixtures Instead of a Database

The app uses local JSON files for the knowledge base, product catalog, reviews, and embeddings. This keeps the assessment easy to run and review without database provisioning. The tradeoff is that catalog updates require a code/data change and deployment rather than an admin workflow.

### In-Memory Retrieval Instead of a Vector Database

Embeddings are generated ahead of time and stored in `data/embeddings.json`. At request time, the app loads the file and performs cosine similarity in memory. This is fast and simple for the current small corpus. A production catalog with thousands of changing records would likely need a managed vector store or database-backed retrieval.

### Client-Stored Profile Instead of Auth

Profile data lives in `localStorage`, which removes auth setup friction for reviewers. The tradeoff is no cross-device persistence, no account recovery, and no server-side user history. A production version would add authentication and persist profiles, chat history, saved products, and checklist state.

### Live LLM Evals Instead of Deterministic Unit Tests Only

The eval suite exercises the real orchestrator and uses LLM judges, so it catches behavior that static assertions miss. The tradeoff is runtime, API cost, network dependency, and some judge variance. The code adds deterministic validation and enrichment where practical to reduce avoidable flakiness.

### Stateless API Routes

The API routes accept the profile and relevant context in each request. This makes the app easier to deploy on serverless infrastructure and avoids session coupling. The tradeoff is larger request payloads if conversation history grows, which a production app would handle with persisted history and summarization.

## API Routes

| Route | Purpose |
| --- | --- |
| `POST /api/brief` | Generate the dashboard stage brief. |
| `POST /api/chat` | Run the classifier plus tool-using chat loop. |
| `POST /api/checklist` | Generate a grounded checklist for a stage. |
| `POST /api/voice` | Transcribe audio and extract shopping-list items. |
| `POST /api/vision` | Analyze a product image for age-fit and safety. |
| `GET /api/catalog` | Return the local product catalog. |
| `GET/POST /api/eval` | Read the latest eval report or run the suite. |

## Deployment

The app is configured for Netlify in `netlify.toml`.

- Build command: `npm run build`
- Publish directory: `.next`
- Required environment variable: `OPENAI_API_KEY`
- `data/embeddings.json` is committed, so Netlify does not need to regenerate embeddings during deploy.

## Limitations

- There is no database or authentication layer in this assessment build.
- User profile data is local to the browser.
- Evals make live OpenAI calls, so they require network access, an API key, and some runtime budget.
- The product catalog and knowledge base are intentionally small local JSON fixtures.
